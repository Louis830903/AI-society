# 你的专属AI助手配置

> 像给小精灵们写的工作手册，让它们更懂你想要什么

## 核心理念：七步思考法则

就像医生看病一样，AI助手在帮你做任何事情之前，都要走完这七步：

### 第一步：停下来想想（暂停反思）
> 想象一下：你要去超市买菜，不是直接冲进去乱拿，而是先想想今天要做什么菜

- 这个任务到底是什么？
- 用户真正想要的是什么？
- 有没有什么我可能误解的地方？

### 第二步：找找线索（收集证据）
> 像侦探一样，先收集所有相关的线索

- 看看相关的代码文件
- 找找类似的例子
- 了解现有的实现方式

### 第三步：挖到根源（根因分析）
> 像拔萝卜一样，要把根都拔出来，不能只处理表面

- 问题的真正原因是什么？
- 不是头痛医头，脚痛医脚
- 找到最根本的问题点

### 第四步：想出办法（方案设计）
> 像画房子的图纸一样，先设计好再动手

- 列出所有可能的解决方案
- 每个方案的优缺点是什么
- 哪个方案最适合当前情况

### 第五步：检查方案（方案审视三步法）
> 像考试前检查试卷一样，确保没有遗漏

1. **完整性检查**：这个方案能解决所有问题吗？
2. **风险评估**：这样做会不会带来新问题？
3. **可行性验证**：现有条件能实现这个方案吗？

### 第六步：动手实施（执行修复）
> 像按照菜谱做菜一样，一步一步来

- 先做最重要的部分
- 每做完一步就检查一下
- 遇到问题及时调整

### 第七步：确认效果（验证结果）
> 像厨师尝味道一样，确保做出来的菜是好吃的

- 测试修改是否有效
- 确认没有引入新问题
- 记录这次经验以便下次参考

## Subagent-Driven 交互模式

> 像派出小精灵分头做事，每个小精灵负责一件事，做完回来汇报

### 工作流程

```
你说一句话
    ↓
主AI理解你的意思
    ↓
分配任务给专门的小精灵（子代理）
    ↓
小精灵完成任务后汇报
    ↓
你审查结果，决定是否继续
```

### 可用的小精灵们

| 小精灵名称 | 负责的工作 | 什么时候叫它 |
|-----------|-----------|-------------|
| 规划师 | 制定计划 | "帮我想想怎么做xxx" |
| 架构师 | 设计系统结构 | "这个功能应该怎么设计" |
| 测试员 | 写测试、检查质量 | "帮我测试一下" |
| 审查员 | 检查代码问题 | "看看这段代码有没有问题" |
| 修复工 | 解决报错和bug | "这里报错了，帮我修" |
| 文档员 | 写说明文档 | "帮我写个说明" |

## 一句话指令示例

> 你只需要像跟朋友说话一样，AI就能理解

### 日常开发
- "帮我加一个登录功能"
- "把这个按钮的颜色改成蓝色"
- "这里的代码太乱了，帮我整理一下"

### 调试修复
- "这里报错了，帮我看看"
- "为什么页面打不开"
- "数据显示不对，帮我查查原因"

### 定时任务（自然语言化）
- "每天早上9点检查一下服务器状态"
- "每隔5分钟刷新一次数据"
- "下周一提醒我更新文档"

## 与AI对话的小技巧

### 表达要清楚
```
❌ "改一下"
✅ "把首页的标题改成'欢迎回来'"
```

### 说明背景
```
❌ "加个按钮"
✅ "在用户信息页面加一个'退出登录'按钮"
```

### 分步骤说
```
❌ "把整个系统重做一遍"
✅ "我们先从用户登录这部分开始，一步步来"
```

---

## 紧急情况处理

### 如果AI做错了
1. 说"停一下"或"先暂停"
2. 说明哪里不对
3. 给出正确的方向

### 如果卡住了
1. 换一种说法描述问题
2. 提供更多背景信息
3. 把大任务拆成小任务

### 如果不确定AI理解了没
1. 让它先说说它的理解
2. 确认后再让它动手
3. 每完成一小步就检查一下
